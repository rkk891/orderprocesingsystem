# Orders API Contract

> **Status:** Planned; no endpoint is implemented. This document owns the HTTP contract.

## 1. Conventions

- Base path: `/api/v1`
- Success media type: `application/json`
- Error media type: `application/problem+json`
- IDs: UUID strings generated by the service
- Timestamps: ISO-8601 UTC instants, for example `2026-07-11T12:30:00Z`
- Enum values are exact uppercase strings.
- Unknown JSON members are rejected with 400. Undocumented query parameters are
  ignored and never change endpoint behavior; only parameters named by an
  endpoint contract are bound.
- No authentication, customer identity, pricing, or product lookup exists in V1.

See [PRD](PRD.md) for behavior, [LLD](LLD.md) for execution, [Data Model](DATA_MODEL.md) for persistence, and [Test Strategy](TEST_STRATEGY.md) for contract coverage.

## 2. Endpoint Summary

| Method | Path | Purpose | Success |
| --- | --- | --- | ---: |
| `POST` | `/api/v1/orders` | Create a multi-item order | 201 |
| `GET` | `/api/v1/orders/{orderId}` | Retrieve full order detail | 200 |
| `GET` | `/api/v1/orders` | List bounded order summaries | 200 |
| `PATCH` | `/api/v1/orders/{orderId}/status` | Advance exactly one fulfilment state | 200 |
| `POST` | `/api/v1/orders/{orderId}/cancel` | Cancel a pending order | 200 |

The background processor has no public endpoint.

## 3. Shared Schemas

### Item input

| Field | Type | Required | Validation |
| --- | --- | ---: | --- |
| `productId` | string | Yes | 1–100 Unicode code points; not blank; must not contain U+0000; exact/case-sensitive uniqueness within request |
| `quantity` | integer | Yes | 1–999 |

### Order detail

| Field | Type | Contract |
| --- | --- | --- |
| `id` | UUID | Server-generated |
| `status` | enum | `PENDING`, `PROCESSING`, `SHIPPED`, `DELIVERED`, or `CANCELLED` |
| `items` | array | 1–100 items in submitted order; each has `productId`, `quantity` |
| `createdAt` | instant | Server-generated UTC timestamp |
| `updatedAt` | instant | UTC timestamp of latest status mutation |

### Order summary

`id`, `status`, `itemCount`, `createdAt`, and `updatedAt`. List responses omit item arrays.

## 4. Create Order

`POST /api/v1/orders`

Request body contains only `items`, an array of 1–100 item inputs. IDs, status, timestamps, item positions, and other fields are server-owned and rejected when supplied.

```json
{
  "items": [
    {"productId": "SKU-RED-01", "quantity": 2},
    {"productId": "SKU-BLUE-02", "quantity": 1}
  ]
}
```

Success is 201 with the order detail and header `Location: /api/v1/orders/{id}`. The initial status is `PENDING`. Duplicate requests create separate orders because create idempotency is outside V1.

Failures: malformed/unknown fields, invalid bounds, empty items, or duplicate product IDs return 400.

## 5. Retrieve Order

`GET /api/v1/orders/{orderId}`

- 200: full order detail.
- 400: path value is not a UUID.
- 404: valid UUID is absent.

## 6. List Orders

`GET /api/v1/orders?status=PENDING&page=0&size=20`

| Query | Required | Default | Validation |
| --- | ---: | --- | --- |
| `status` | No | all statuses | Exact known enum, including `CANCELLED` |
| `page` | No | `0` | Integer at least 0 |
| `size` | No | `20` | Integer from 1 through 100 |

Sorting is fixed to `createdAt DESC`, then `id DESC`; clients cannot override it.
A `sort` or other undocumented query parameter is ignored. Each documented
parameter may occur at most once; repetition is invalid input. “All orders”
means every order is traversable across pages for a stable dataset.

Response fields:

| Field | Type | Contract |
| --- | --- | --- |
| `content` | array | Zero or more order summaries |
| `page` | integer | Current zero-based page |
| `size` | integer | Requested page size |
| `totalElements` | integer | Matching order count |
| `totalPages` | integer | Page count for current size |
| `first` | boolean | Current page is first |
| `last` | boolean | Current page is last |

Invalid status/page/size returns 400. A valid page with no matches returns 200 with empty `content`.

## 7. Advance Status

`PATCH /api/v1/orders/{orderId}/status`

```json
{"status": "SHIPPED"}
```

The body contains exactly one known `status`. Only targets `PROCESSING`, `SHIPPED`, and `DELIVERED` can succeed; the persisted predecessor must be respectively `PENDING`, `PROCESSING`, or `SHIPPED`.

- 200: full updated order detail.
- 400: malformed UUID/body/enum or unknown field.
- 404: valid ID absent.
- 409: skip, reversal, self-transition, generic `CANCELLED`, terminal-state update, or lost concurrent race.

For the known targets `PENDING` and `CANCELLED`, which are never valid manual
targets, target validation takes precedence and returns 409 without using order
existence. Other known targets reach the conditional update, where a missing ID
returns 404.

## 8. Cancel Order

`POST /api/v1/orders/{orderId}/cancel`

No request body is accepted. The body must be absent or zero bytes; JSON `null`,
`{}`, whitespace, and any other non-zero-length payload are invalid. A
`Content-Type` header without a body does not itself make the request invalid.

- 200: full detail with `CANCELLED`.
- 400: malformed UUID or non-empty/malformed body.
- 404: valid ID absent.
- 409: order exists but is not `PENDING`, including an already-cancelled order or a scheduler race lost by cancellation.

Cancellation is a state transition, never deletion.

## 9. Problem Details

Errors use an RFC 9457 Problem Details object with stable application
extensions:

```json
{
  "type": "urn:problem:order-state-conflict",
  "title": "Order state conflict",
  "status": 409,
  "detail": "The order cannot make the requested transition.",
  "instance": "/api/v1/orders/7bd36f1f-90d4-41dd-8a89-9aa622dfc0ad/status",
  "code": "ORDER_STATE_CONFLICT",
  "traceId": "8d4e9c1a",
  "violations": []
}
```

Every problem response contains all eight top-level members shown above:

| Member | Contract |
| --- | --- |
| `type` | Stable URI below; clients use this or `code` as the machine identifier. |
| `title` | Stable short title below; clients must not parse it. |
| `status` | HTTP status repeated as an integer. |
| `detail` | Stable safe explanation below; never contains SQL, exception, or secret data. |
| `instance` | Request path for this occurrence, without scheme, authority, or sensitive query values. |
| `code` | Stable application code below. |
| `traceId` | Non-blank request correlation value; format is intentionally opaque. |
| `violations` | Always an array; empty unless one or more client-input locations can be identified. |

| Code | Type | Title | HTTP | Detail |
| --- | --- | --- | ---: | --- |
| `INVALID_REQUEST` | `urn:problem:invalid-request` | `Invalid request` | 400 | `The request is malformed or violates validation rules.` |
| `ORDER_NOT_FOUND` | `urn:problem:order-not-found` | `Order not found` | 404 | `The requested order does not exist.` |
| `ORDER_STATE_CONFLICT` | `urn:problem:order-state-conflict` | `Order state conflict` | 409 | `The order cannot make the requested transition.` |
| `INTERNAL_ERROR` | `urn:problem:internal-error` | `Internal error` | 500 | `An unexpected error occurred.` |
| `METHOD_NOT_ALLOWED` | `urn:problem:method-not-allowed` | `Method not allowed` | 405 | `The HTTP method is not supported for this resource.` |
| `NOT_ACCEPTABLE` | `urn:problem:not-acceptable` | `Not acceptable` | 406 | `The requested response representation is not available.` |
| `UNSUPPORTED_MEDIA_TYPE` | `urn:problem:unsupported-media-type` | `Unsupported media type` | 415 | `The request media type is not supported.` |

A violation contains exactly `field` and `message`, both strings. Body fields use
JSON-style paths such as `items[0].quantity`; query and path parameters use their
parameter name, such as `size` or `orderId`; `$` identifies the whole request
when no narrower location exists. Violations are sorted by `field`, then
`message`. Messages are stable human-readable text, but clients branch on
`code`/`field`, not message wording. Malformed JSON can therefore return an empty
array when no reliable field is available. Database details, stack traces, and
secrets never appear.

## 10. Retry and Compatibility Semantics

- GET operations are safe and read-only.
- Repeating create creates another order; clients must not assume an idempotency key.
- Repeating a successful status update or cancel returns 409 because the precondition no longer holds.
- The internal pending processor is idempotent: an immediate rerun updates zero rows.
- Backward-incompatible HTTP changes require a new API version and coordinated PRD/API/test updates.

The service generates one opaque trace ID for every request, returns it as
`X-Trace-Id`, and uses the same value in Problem Details and safe request-outcome
logs. Client-provided trace IDs are not trusted as the server identifier.

Create idempotency keys are deliberately deferred, not overlooked. Mature
commerce APIs commonly persist a client key with the first mutation result so a
network retry does not create a second resource. V1 has no checkout/payment side
effect and the assignment does not require safe create retries, so adding the
storage, request-fingerprint, expiry, and replay contract would be speculative.
Revisit before clients are allowed to retry order creation automatically.
